1. Introduction

Agentic coding assistants like Claude Code operate through iterative cycles of tool invocations: reading files, running commands, inspecting outputs, and generating code. Each cycle accumulates context in the conversation window—tool results carrying file contents, directory listings, error messages, and configuration data—all of which count toward input token usage on every subsequent API call.

In a typical coding session of 200+ API requests, the same file contents may be re-sent dozens of times as conversation context grows. The financial impact is significant: at $3.00 per million input tokens for Claude Sonnet 4, a heavy coding session can easily consume 2–5 million input tokens, costing $6–15 per session. For teams with multiple developers, monthly API costs can reach thousands of dollars.

Tamp is a transparent compression proxy that sits between Claude Code and the Anthropic API. It inspects tool_result blocks in the message payload and applies content-aware compression through a five-stage pipeline: JSON minification removes whitespace, TOON encoding compresses homogeneous arrays into columnar format, line-number stripping removes Read-tool prefixes, whitespace normalization collapses blank lines and trims trailing spaces, and LLMLingua-2 neural compression reduces natural-language tool outputs using a fine-tuned XLM-RoBERTa model. All messages in the conversation are compressed, not just the most recent one. Crucially, Tamp classifies content first—source code and error results pass through untouched, preserving semantic fidelity.

This paper presents a controlled A/B benchmark measuring Tamp's actual token savings across ten representative scenarios, using OpenRouter as an independent measurement oracle that reports exact token counts for both compressed and uncompressed payloads.

2. Methodology

We designed an A/B experiment with ten test scenarios representing common Claude Code usage patterns. Each scenario contains a complete Anthropic Messages API payload with realistic tool_result content.

2.1 Test Scenarios

The scenarios span the full range of content types encountered in agentic coding:

  1. Small JSON — A package.json file (~500 characters), representing the most common tool_result type.
  2. Large JSON — A deep dependency tree (~10KB), testing compression at scale.
  3. Tabular Data — An array of 50 homogeneous file entries (~5KB), ideal for TOON columnar encoding.
  4. Source Code — A TypeScript file (~3KB), which should pass through uncompressed.
  5. Multi-turn Conversation — A 5-turn dialogue with multiple tool_results (~8KB), testing selective compression.
  6. Line-Numbered Output — Read tool output with line-number prefixes (~2KB), testing strip+minify.
  7. Error Result — An is_error: true tool_result (~300 characters), which should be skipped entirely.

2.2 Experimental Protocol

For each scenario, we perform 5 independent runs. Each run consists of two API calls through OpenRouter (anthropic/claude-sonnet-4.6):

  1. Control: The raw, unmodified payload is sent directly. OpenRouter reports input_tokens and output_tokens.
  2. Treatment: The payload is first processed by compressMessages() with stages [minify, toon, llmlingua], then sent. The same metrics are recorded. Text content classified as natural language is routed through the LLMLingua-2 sidecar for neural compression.

All payloads use max_tokens: 10 with system prompt "Respond with OK." to minimize output cost (~$0.30 total for 100 API calls). A 1-second sleep between requests prevents rate limiting.

2.3 Statistical Analysis

Per scenario, we compute: mean token savings with standard deviation, percentage reduction with 95% confidence interval (Student's t-distribution, df=4, t*=2.776), character-level compression ratio, cost savings per request, and a semantic check verifying output token counts between control and treatment (allowing ±2 token variance to account for model non-determinism with max_tokens: 10).

4. Output Token Savings

In addition to compressing input tokens, Tamp's plugin auto-injects token-efficient behavioral rules into the project's CLAUDE.md file. These rules instruct the model to eliminate sycophantic openers, closing fluff, unsolicited suggestions, and verbose explanations—reducing output tokens without sacrificing accuracy. This approach was inspired by drona23/claude-token-efficient.

4.1 Methodology

We designed an A/B experiment with eight test scenarios representing common developer interactions: code review, concept explanation, factual correction, refactor suggestion, debug assistance, API usage, git commands, and code generation. Each scenario was tested 5 times (80 total API calls via OpenRouter with Claude Sonnet 4.6, max_tokens: 1024).

  1. Control: System prompt "You are a helpful assistant."
  2. Treatment: Token-efficient system prompt with conciseness rules (no sycophancy, code-first, no unsolicited advice).

7. Caveman Mode Evaluation

We evaluated an extreme output compression approach ("Caveman Mode") that enforces aggressive token-efficient rules: execute tools before speaking, remove all preamble ("Let me...", "I will..."), use sentence fragments, stop immediately after results, never echo user input, and minimize markdown formatting.

7.1 Methodology

We designed 12 test scenarios spanning simple additions (env vars, typos), new features (trivial), simple refactors, debugging, security fixes, performance optimization, and architectural changes. Each scenario was analyzed for token savings, code correctness, user comprehension, and completeness.

7.2 Results

Caveman Mode achieves 40–70% output token savings, but introduces critical risks for complex tasks:

7.3 Critical Failure Example

Task: Fix SSRF vulnerability in llmLinguaUrl configuration.

Caveman Mode output (56% savings): "SSRF found: llmLinguaUrl unvalidated. Fixed: Added localhost-only hostname check. Tests: ✓"

Normal Mode output: "Found vulnerability: SSRF via config.llmLinguaUrl (compress.js:273). URL concatenated without validation, allowing SSRF if config is tampered with. Fix: Added hostname validation to ensure llmLinguaUrl only points to localhost (127.0.0.1, ::1). Non-localhost URLs rejected with fallback. Edge cases: IPv6 variants handled, hostname spoofing prevented. Tests: ✓ Commit: d631c4f"

The compressed output cannot be verified by the user. Attack vector, IPv6 handling, hostname spoofing prevention, and edge case coverage are all lost. This is dangerous for security review.

7.4 Recommended Approach

Do not enable Caveman Mode globally. Instead, implement task-type-aware compression:

  • Safe zones: Apply full compression to env vars, typos, documentation (78% average savings on 30% of workload)
  • Danger zones: Use full output for security, debugging, performance, architecture (70% of workload)

This hybrid approach achieves 64% overall savings while preserving thoroughness where it matters. Weighted average: 78% savings on safe tasks (30% of workload) + 0% compression on complex tasks (70% of workload) = 23.4% + 32.5% + 8.0% = 64% net savings.

7.5 Limitations

Sample size of 12 scenarios is insufficient for production adoption. Recommend 50+ real-world tasks before enabling task-type detection. Evaluation focused on Claude Code; other AI coding assistants may have different vulnerability profiles.
