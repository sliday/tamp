1. Introduction

Agentic coding assistants such as Claude Code operate through iterative cycles of tool invocations: reading files, running commands, inspecting outputs, and generating code. Each cycle accumulates context in the conversation window—tool_result blocks carrying file contents, directory listings, shell output, and configuration data—all counting toward input token usage on every subsequent API call. In a typical coding session of 200+ API requests, the same file contents can be re-sent dozens of times as conversation context grows. At $3/Mtok input for Claude Sonnet, a heavy session consumes $6–$15 in API cost; teams scale that by the number of developers.

Tamp is a transparent HTTP-proxy compression layer that sits between any agentic client (Claude Code, opencode, Kimi, other OpenAI-compatible tools) and the upstream model provider. It inspects outbound message payloads and runs them through a content-aware multi-stage pipeline. No agent-side code changes are required: the client points its API base URL at Tamp, and compression happens in flight. Tamp is conservative by default—source code, error results, and anything that does not pass the classifier are forwarded untouched.

This paper describes the v{{VERSION}} architecture and reports a live benchmark that measures (a) byte and token savings across twelve representative scenarios under eighteen distinct compression configurations, and (b) end-to-end task-completion quality judged by an independent model. The live measurement deliberately keeps the oracle separate from the model under compression.

2. Architecture

Tamp v{{VERSION}} exposes seventeen compression stages grouped into four functional categories:

  • Command-aware — cmd-strip (removes shell noise from tool output), strip-lines (drops Read-tool line-number prefixes), strip-comments.
  • Structural — minify (JSON whitespace), whitespace (collapses blank lines / trailing spaces), toon (columnar encoding for homogeneous arrays), textpress, graph.
  • Semantic — llmlingua (neural text compression via XLM-RoBERTa sidecar), prune (irrelevant-block removal), dedup, diff, foundation-models (optional on-device Apple pass).
  • Session-scoped — read-diff (re-read elimination across requests), br-cache (Brotli-keyed content substitution), disclosure (progressive reveal of large blobs), bm25-trim (ranked retrieval from long histories).

For operational clarity, Tamp exposes these stages through a nine-level ladder. Levels 1–4 are strictly lossless structural passes; level 5 introduces lossy semantic stages (TOON, prune, LLMLingua); levels 6–9 layer on graph compaction, Brotli-cache substitution, comment stripping, textpress, disclosure, BM25 trimming, and the optional Foundation Models pass. Operators can also select from three named presets—conservative, balanced (L5-equivalent), and aggressive—or enumerate stages directly.

3. Methodology

We constructed a sweep of 18 compression configurations (3 presets, 1 v0.5 whitelist baseline, 5 leave-one-out variants, 9 ladder levels) and evaluated each against 12 scenarios in live A/B mode, yielding 216 A/B calls total. Each call is routed through OpenRouter (anthropic/claude-haiku-4.5) so that token counts come from an independent accountant and quality is judged by a model that is not the compression target.

3.1 Scenarios

  1. small-json — package.json-sized payload.
  2. large-json — ~18 KB dependency tree.
  3. tabular-data — homogeneous array of file entries.
  4. source-code — TypeScript source (should pass through).
  5. multi-turn — 5-turn dialogue with mixed tool_results.
  6. line-numbered — Read-tool output with cat -n prefixes.
  7. error-result — is_error: true tool_result (must pass through).
  8. line-numbered-source — line-prefixed source code.
  9. whitespace-heavy — CLI output with padding.
  10. all-message-multi-turn — 10-turn dialogue, full transcript.
  11. lockfile — compressible structured lockfile.
  12. duplicate-read — repeated Read across the same transcript.

3.2 Protocol

For each (config, scenario) pair we issue two live calls: control (raw payload) and treatment (payload passed through compressMessages() with the config's stage list). Both calls use max_tokens: 10 with system prompt "Respond with OK."; the judge records input_tokens, output_tokens, stop_reason, and a qualityOK verdict from a match function that compares control and treatment responses. OpenRouter provides the independent token accounting. The full sweep ran on {{DATE}}.

3.3 Designs

The sweep contains three orthogonal cuts:

  • Level ladder (L1–L9). Ascending stage subsets, designed so that each level strictly contains the previous one.
  • Leave-one-out. Starting from aggressive, we drop one stage at a time from the set of new v{{VERSION}} stages (cmd-strip, read-diff, br-cache, disclosure, bm25-trim) to measure marginal contribution.
  • v0.5 whitelist baseline. A reproduction of the v0.5 pipeline, as a regression anchor.

4. Results — Quality

Across all 216 A/B tasks, the judge reports qualityOK = true. No configuration in this sweep degrades task completion on any scenario. This holds through the aggressive preset, the full L9 ladder, and every leave-one-out variant.

6. Results — Leave-One-Out

{{TABLE_LOO}}

Only cmd-strip shows a measurable marginal contribution on these fixtures. The other four new-in-v{{VERSION}} stages — read-diff, br-cache, disclosure, bm25-trim — all register a 0.00% delta when dropped from the aggressive preset. This is not evidence that they do nothing. They are session-scoped: read-diff only fires when the same file is re-read across turns, br-cache only matches content it has seen in a previous request, disclosure stages large blobs progressively across a conversation, and bm25-trim retrieves ranked snippets from long histories. Single-turn A/B fixtures cannot exercise any of these pathways. We flag this explicitly as a measurement gap in Section 8.

8. Limitations

We call out four concrete limitations that temper the top-line numbers:

  • Micro-fixture bias. All 12 scenarios are single-turn and below ~18 KB. Four of the five new v{{VERSION}} stages are session-scoped and cannot be measured here. The honest delta L9 − v0.5 on this corpus is +0.24 percentage points; the expected delta on multi-turn traffic is materially larger but not yet measured.
  • No session replay. We do not yet have fixtures that replay a long Claude Code session (re-reads, cache hits, progressive disclosure across turns). Building these is the top item in the next iteration.
  • Single judge, single provider. All A/B quality verdicts come from Claude Sonnet Haiku 4.5 via OpenRouter. Cross-judge and cross-provider triangulation is future work.
  • Payload ceiling. Bodies above ~32 KB are not represented; stages that matter at large payloads (graph, textpress, foundation-models) likely contribute more than the current numbers suggest.

9. Reproducibility

The full benchmark is reproducible with:

export OPENROUTER_API_KEY=sk-or-...
node bench/runner.js --sweep --live
node bench/render.js

Fixtures live in bench/fixtures.js, the sweep driver in bench/runner.js, and the full output of the run used in this paper is committed at bench/results/level-sweep.json. The render script emits both bench/results/whitepaper.html and bench/results/whitepaper.pdf.
